## Что строим

Бот принимает ссылки и медиафайлы. Транскрибирует, сохраняет в базу, форвардит в приватный архив-канал. Дальше — генерирует контент под твой голос (Модуль 2, отдельно).

**Что работает сейчас:**
- YouTube видео — транскрипт через youtube-transcript-api или Groq Whisper
- TikTok видео — транскрипт через Groq Whisper (аудио через прокси)
- Фото и PDF — текст через Google Vision OCR

**Что не работает (и почему):**
- Instagram (reels и карусели) — Instagram блокирует все автоматические запросы без авторизации. Это не баг, это их политика. Без аккаунта — никак.
- TikTok карусели (посты со слайдами) — yt-dlp не поддерживает `/photo/` URL-формат.

## Шаг 4.1 — database/db.py: таблица library_content

Первый файл кода — база данных.

Смотри как работает субагент-driven development:
Claude Code запускает свежий агент на задачу "создай db.py",

Три функции: `init_db()`, `insert_content()`, `get_content_by_id()`.

## Шаг 4.2 — services/vision.py: OCR и PDF

`services/vision.py` — две функции:

- `extract_text_from_image(bytes)` → строка текста через Google Vision API
- `extract_text_from_pdf(bytes)` → текст всех страниц через pdfplumber

Никакого сохранения файлов. Принял байты — вернул текст. Чисто.

```python
from content_bot.services.vision import extract_text_from_image, extract_text_from_pdf

text = extract_text_from_image(image_bytes)
text = extract_text_from_pdf(pdf_bytes)
```

Google Vision API подключается через API-ключ из переменной `GOOGLE_VISION_API_KEY`.
Ключ взял из шага 2 — тот же что в основном боте.

Если застрял — скриншот в Claude Code.

## Шаг 4.3 — services/content_processor.py: yt-dlp

`services/content_processor.py` — три функции:

- `detect_url_type(url)` → `{platform, content_type}` или `None`
- `parse_vtt_text(vtt_content)` → чистый текст без таймкодов
- `process_url(url)` → `ProcessedContent` с транскриптом

Никакого LLM на этом этапе. Бот — механический сборщик.
Транскрипт извлекается в три попытки:

1. **youtube-transcript-api** — самый быстрый путь для YouTube, без скачивания
2. **Groq Whisper** — скачивает аудио через yt-dlp (mp3, ~64kbps), отправляет в Whisper API
3. **yt-dlp субтитры** — последний резерв, `.vtt` файл без аудио

Для TikTok и Instagram: сначала субтитры через yt-dlp, потом Groq Whisper.
Все yt-dlp вызовы идут через residential proxy (WEBSHARE_PROXY_URL) —
без этого Render'овский datacenter IP блокируется YouTube и TikTok.

```bash
# Что делает yt-dlp под капотом:
yt-dlp --write-auto-sub --sub-lang ru,en --skip-download \
  --output /tmp/%(id)s https://youtube.com/watch?v=...
```

Временный файл субтитров удаляется сразу после чтения. Никакого хранения медиа.

Если застрял — скриншот в Claude Code.

## История: три фикса и один диагноз

Бот задеплоился. Карточки сохраняются. Всё выглядит хорошо — но транскрипта нет.
Поле всегда пустое. Ни у YouTube, ни у TikTok.

**Фикс 1.** `youtube-transcript-api` — в версии 0.7+ убрали метод `get_transcript` у класса.
Нашел, переписал на инстанс-метод. Задеплоил. Транскрипта нет.

**Фикс 2.** Добавил Groq Whisper как fallback: скачиваем аудио через yt-dlp, отправляем в Whisper API.
Задеплоил. Транскрипта нет. При этом логи показывают 6 секунд на обработку 34-минутного видео.
Что-то не так на уровне глубже кода.

**Фикс 3.** Добавил Groq Whisper fallback для TikTok и Instagram тоже — вдруг проблема в ветке.
Задеплоил. Транскрипта нет.

Три правильных фикса. Ни один не работает.

---

Вместо Фикса 4 — включил режим мудреца:

```bash
/brainstorming решить проблему транскрипции в content-bot
```

Claude не предложил следующий патч. Задал вопрос: GROQ_API_KEY добавлен на Render?
Да, добавлен. Тогда почему 6 секунд на 34-минутное видео?

Потому что yt-dlp не скачивает аудио. Он падает почти моментально с ошибкой.
YouTube и TikTok детектируют datacenter IP и блокируют на уровне сети.
Render — shared datacenter. Все три фикса атаковали не ту проблему.

**Решение:** residential proxy. Один env var (`WEBSHARE_PROXY_URL`), все платформы фиксируются сразу.

---

Это и есть точка применения режима мудреца — не когда застрял после первой попытки,
а когда три правильных фикса не работают и ты готов писать четвертый.
Именно тогда нужно выйти из режима исполнения и начать диагностировать.

**Итог:** прокси сработал. YouTube и TikTok-видео теперь получают полный транскрипт.

---

После этого — следующий раунд. Instagram. Карусели (слайды с инфографикой) и Reels.

yt-dlp скачивает видео через прокси — но Instagram это не YouTube. У него нет публичного API для контента. Любой запрос без авторизованного аккаунта возвращает "login required". Прокси не помогает — это не проблема IP, это политика закрытого сада.

Код для instaloader (Python-библиотека с логином через username + password) написан и задеплоен. Нужно только добавить два env var на Render — и Instagram заработает. Но это throwaway аккаунт, которого пока нет.

**Решение:** принять ограничение сейчас, добавить в бэклог.

Иногда правильное инженерное решение — это не решить задачу, а признать где проходит граница. Бот делает то, для чего создан: YouTube, TikTok-видео, фото, PDF. Instagram — в бэклоге, под конкретный тикет с конкретным условием (throwaway аккаунт).

## Шаг 5 — Деплой на Render

Последний шаг — запустить бота в облаке.

**render.com → New → Background Worker**

> Важно: выбирай именно **Background Worker**, не Web Service.
> Web Service ждет HTTP-сервер на порту — polling-бот его не запускает, и Render убивает процесс.

1. Connect Repository → выбери `content-bot`
2. Build Command: `pip install -r requirements.txt`
3. Start Command: `python3 bot.py`

**Environment Variables** — добавляешь все переменные из `.env.example`:

```
BOT_TOKEN             = токен из BotFather (Шаг 2)
DATABASE_URL          = External URL из Render PostgreSQL (Шаг 2)
GOOGLE_VISION_API_KEY = ключ из Google Cloud (Шаг 2)
LIBRARY_CHANNEL_ID    = ID приватного канала
GROQ_API_KEY          = ключ из console.groq.com (бесплатно)
WEBSHARE_PROXY_URL    = rotating endpoint из webshare.io (бесплатный тир)
```

**Deploy** → ждешь зеленый статус.

Проверка: зайди в Telegram → кинь боту TikTok-ссылку → должен ответить `✅ Сохранено #1`

## Модуль 2 — откуда взялся Google Sheets

Модуль 1 работает. Транскрипты сохраняются в PostgreSQL, карточки летят в архив-канал.

Но есть проблема. Библиотека растет, а посмотреть что там — нельзя. Залезть в базу я не собираюсь. Видео без названий, транскрипты без контекста, непонятно что стоит адаптировать а что нет. Слепой архив.

Сел в режим мудреца и начал думать что именно нужно в Модуле 2.

Оригинальный план был простой: `/generate` → бот спрашивает платформу → генерирует скрипт. Но это решает не ту проблему. Проблема не в генерации — проблема в том что я не вижу что у меня накоплено и не могу принять решение что адаптировать.

Правильный вопрос: где я хочу управлять библиотекой? Ответ очевидный — Google Sheets. Вижу всё, фильтрую, ставлю галочки, ничего не ломается.

Так появилась новая архитектура.

При сохранении контента бот теперь делает три вещи вместо одного: сохраняет транскрипт в базу, запускает Gemini Flash анализ (хук, структура, потенциал адаптации), и пишет всё это в Google Sheets. Я открываю таблицу и вижу живую библиотеку с разбором каждого материала.

Когда решил что что-то стоит адаптировать — меняю статус на "одобрено" и ставлю галочки напротив платформ. Бот поллит таблицу каждые 5 минут. Находит одобренные строки, генерирует скрипты через Gemini Flash в моем голосе, заполняет колонки. Открываю Sheets — скрипты готовы.

Два решения которые не очевидны сразу:

Первое — Gemini Flash для всего. Не Claude, не OpenAI. Gemini Flash бесплатный на 1500 запросов в день. При моем объеме это бесконечность. Качество для генерации постов достаточное, особенно с системным промптом и тремя примерами моих постов как few-shot.

Второе — polling вместо webhook. Можно было сделать триггер в Google Apps Script который вызывает Render при изменении статуса. Но это второй сервис, вебхук, дополнительная сложность. Polling каждые 5 минут живет в том же Background Worker — одна кодовая база, один деплой, нулевая дополнительная стоимость.

## Модуль 2 — первая версия и почему она не взлетела

Реализовал Модуль 2 как и планировал: Gemini Flash анализирует транскрипт при инжесте, пишет в Sheets, поллер каждые 5 минут находит одобренные строки и генерирует скрипты.

Задеплоил. Отправил YouTube-ссылку. Транскрипта нет. Анализа нет.

Начал копать. Оказалось — не одна проблема, а три одновременно.

**Проблема 1 — YouTube транскрипт.** youtube-transcript-api работает локально, но Render — datacenter. YouTube видит datacenter IP и блокирует. Fallback через yt-dlp + Groq Whisper: yt-dlp тоже блокируется. Добавил прокси для yt-dlp — аудио скачивается, но в логах нет никаких ошибок, нет ответа, просто тишина. Без debug логирования не видно что происходит.

**Проблема 2 — Gemini квота.** Включил debug логирование. Смотрю логи — Gemini возвращает 429: `GenerateRequestsPerDayPerProjectPerModel-FreeTier`. Дневной лимит бесплатного tier исчерпан. Не за месяц, не за неделю — за один день тестирования. Retry на 429 добавлен, но дневной лимит retry не обходит.

**Проблема 3 — стоимость масштабирования.** Чтобы Gemini нормально работал нужен биллинг. Чтобы Groq Whisper работал на Render без прокси-проблем — нужен нормальный прокси (платный). Чтобы генерация скриптов была качественной — Groq Llama 70B нормальный, но все равно API. Получается стек из трех платных сервисов поверх Render, который и так платный.

Сел в режим мудреца и переосмыслил.

## Два подхода к Модулю 2

Вместо того чтобы чинить по одной — поставил вопрос иначе: а нужен ли вообще Модуль 2 в боте?

Выкристаллизовались два принципиально разных подхода.

**Версия A — чинить то что есть.** Оставить Модуль 2 в боте. Заменить Gemini на Groq Llama 3.3 70B (квота выше), переписать анализ под комната-идей формат, добавить генерацию 8 форматов (YouTube long/short, TikTok, Shorts, LinkedIn long/short, Telegram long/short). Поллер убрать — генерировать всё сразу при инжесте.

Плюсы: бот всегда включен, ссылки кидаешь с телефона, всё в облаке.
Минусы: Groq Llama 70B — это все равно API с лимитами. Качество анализа хуже чем Claude. YouTube блокировка на Render не решена. И главное — по хорошему для нормальной работы нужны дополнительные платные API: прокси для стабильной работы на Render, биллинг на Groq при росте объема, возможно отдельный API для транскрипции. Стоимость растет с объемом.

**Версия B — `idea-room` скил.** Глобальный Claude Code скил, запускается локально с мака. Принимает URL или файл любого формата: видео, аудио, картинка, PDF, вебсайт. YouTube транскрибирует через youtube-transcript-api (работает с мака без блокировок). TikTok и Instagram скачивает через yt-dlp и транскрибирует через mlx-whisper — локально, бесплатно, на Apple Silicon. Сайты и статьи читает через WebFetch. Картинки и PDF обрабатывает Claude Vision нативно. Анализирует Claude в текущей сессии — не Llama, не Groq, а Claude. Сохраняет markdown в `комната-идей/`, обновляет Library Sheets и пишет все 8 форматов в отдельный Content Calendar Sheets с автоматическим расписанием публикаций.

Плюсы: полностью бесплатно (кроме Claude Code подписки которая уже есть), качество анализа лучше — Claude вместо Llama, файлы локально на диске, никаких datacenter IP проблем, никаких прокси.
Минусы: нужен компьютер рядом, нет фонового сбора 24/7.

На бумаге Версия B выглядела как победа. Бесплатно, качество выше, никаких datacenter IP проблем.

Начал реализацию. И упёрся в стену которую не видел в анализе.

`claude -p` в headless режиме запрашивает разрешение на запись файлов. Интерактивный диалог — прямо в середине автоматизированного пайплайна. `--dangerously-skip-permissions` не помогает. Скил запускается, Claude работает, но markdown-файл в `комната-идей/` не создается. Без файла нет истории, нет Content Calendar, нет смысла.

Это не баг который можно запатчить. Это ограничение архитектуры.

## Версия C — и почему это была правильная ошибка

Вернулся к исходному вопросу: что я на самом деле хочу?

Хочу скинуть ссылку с телефона в 23:00 и утром найти готовые скрипты. Без открытого терминала, без запущенного мака, без ручных команд. Полностью автономно.

## История: когда JSON выглядит как победа но это не JSON

Задеплоил Версию C. Выставил статус "одобрено", поставил галочки. Поллер сработал. В Content Calendar Sheet появились строки.

Открываю — в колонке Контент сырой JSON с markdown-фенсингом:

```
```json
{
  "hook": "...",
  "content": "..."
}
```
```

Колонка Хук пустая.

Системный промпт прямо говорит Claude: "отвечай валидным JSON без markdown-блоков". Claude читает инструкцию и всё равно добавляет ` ```json ``` `. Это известный паттерн — LLM-ы часто делают это когда у них есть склонность к форматированию.

Фикс простой: зачищать фенсинг перед `json.loads()`:

```python
raw = re.sub(r"^```(?:json)?\s*", "", raw)
raw = re.sub(r"\s*```$", "", raw).strip()
data = json.loads(raw)
```

Не менять промпт. Не добавлять retry. Просто принять что модель иногда добавляет фенсинг — и убирать его на стороне кода.

Запушил. Проверил. Хук и Контент в правильных колонках.

## Модуль 4 — Автопубликация: от черновика до поста в канале

Скрипты готовы. Дата в календаре. Но публиковать всё равно нужно вручную: открываю Sheets, копирую текст, иду в Telegram, создаю пост. Каждый раз.

Это не то что я хотел строить.

Финальный шаг: бот сам публикует в канал в нужное время.

**Расписание через env var.** Не хардкодить дни и время в код — это закрытие гибкости. Одна переменная в Render:

```
POSTING_SCHEDULE=TUE,THU,SAT@18:00
```

`scheduler.py` парсит это в список дней недели и время. Функция `next_publish_date()` принимает уже занятые даты (из Library Sheet) и возвращает ближайшую свободную. Два поста не попадают на один день автоматически.

**Image card.** Пост с картинкой собирает охват. Пост без картинки теряется в ленте.

`image_card.py` — Pillow, 1080×1080. Тёмный фон как у GitHub (#0d1117). Хук крупным шрифтом по центру. Водяной знак `@headlessaimode` внизу. Шрифт — PIL default, работает без установки дополнительных пакетов на Render.

Карточка генерируется каждый раз перед публикацией из хука поста. Нигде не хранится.

**Publisher.** PTB job каждые 60 секунд. `get_due_posts()` смотрит в Library Sheet — есть ли строки где дата публикации ≤ сегодня и статус ≠ опубликовано. Если есть — генерирует image card, отправляет в канал.

Одна проблема которую не видел до реализации: Telegram ограничивает caption (подпись к фото) в 1024 символа. Пост с hook + контентом часто длиннее. Решение:

```python
if len(caption) <= 1024:
    await bot.send_photo(channel_id, photo=card, caption=caption)
else:
    await bot.send_photo(channel_id, photo=card, caption=hook)
    await bot.send_message(channel_id, text=content)
```

Если текст не влезает — фото с хуком, потом отдельное сообщение с полным контентом. Два сообщения вместо одного обрезанного.

После публикации: поллер ставит ✓ в колонку "Опубликовано" в Library Sheet. Строка больше не попадает в очередь.

---

**Что нужно сделать один раз:**

1. Добавить content-bot как admin канала @headlessaimode (права "Post Messages")
2. Получить numeric ID канала через @userinfobot (выглядит как `-100XXXXXXXXX`)
3. Добавить на Render:

```
POSTING_SCHEDULE=TUE,THU,SAT@18:00
TELEGRAM_CHANNEL_ID=-100XXXXXXXXX
ANTHROPIC_API_KEY=sk-ant-...
```

4. Добавить заголовок `Опубликовано ✓` вручную в ячейку S1 Library Sheet
5. Deploy → проверить логи: должна быть строка `Publisher scheduled: every 60s`

После этого: одобрил строку в Sheets → поллер сгенерировал скрипты и назначил дату → publisher в нужный день отправил пост в канал. Без открытого мака, без ручного копипаста.

---

**Полный flow сейчас:**

1. Кидаю ссылку боту с телефона
2. Бот транскрибирует, анализирует через Claude Haiku, пишет в Library Sheet
3. Открываю Sheets, ставлю статус "одобрено" и галочки платформ

## Модуль 5 — Google Drive: посты как живые документы

Система работает. Скрипты генерируются, даты назначаются, посты уходят в канал по расписанию.

Но есть трение. Хочу отредактировать пост перед публикацией — открываю Sheets, нахожу нужную ячейку, правлю там. Это неудобно: ячейка маленькая, форматирование не видно, нет истории изменений. Google Docs для этого сделан, Sheets — нет.

Решение простое: пусть бот сам создает Google Doc для каждого поста. Я открываю папку в Drive, вижу все черновики, правлю в нормальном редакторе. А бот каждые 5 минут тянет изменения обратно в Sheets — так publisher всегда берет актуальную версию.

**Структура папок в Drive:**

```
Headless AI Mode — Content/
  Telegram/
  LinkedIn/
  TikTok/
  YouTube/
```

Папки создаются автоматически при первом запуске поллера — не нужно создавать вручную.

**Как работает:**

1. Поллер нашел одобренную строку, сгенерировал скрипты
2. Для каждой платформы создается Google Doc в соответствующей папке — название Doc это SEO-заголовок поста + дата
3. Doc ID сохраняется в Library Sheet (колонка "Doc IDs" как JSON)
4. Doc sync job каждые 5 минут читает все Docs с незакрытыми постами
5. Если текст в Doc отличается от текста в Sheets — обновляет Sheets
6. Publisher берет из Sheets что там есть на момент публикации

Полный flow теперь выглядит так: одобрил строку в Sheets → поллер сгенерировал скрипты → создал Docs в Drive → назначил дату → создал ивент в Calendar → в назначенное время publisher опубликовал в канал.

**Что нужно сделать один раз:**

1. Включить Google Drive API и Google Docs API в Cloud Console (тот же проект что и Sheets)
2. Создать папку "Headless AI Mode — Content" в Drive
3. Расшарить папку с email сервис-аккаунта (редактор) — email берешь из поля `client_email` в JSON `GOOGLE_SHEETS_CREDENTIALS`
4. Скопировать ID папки из URL: `drive.google.com/drive/folders/ВОТ_ЭТО`
5. Добавить на Render: `DRIVE_CONTENT_FOLDER_ID=<ID папки>`

После деплоя — следующий одобренный пост создаст Docs автоматически.
